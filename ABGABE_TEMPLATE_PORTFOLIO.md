# Abgabe-Template für Portfolioarbeiten

Dieses Template bietet eine kompakte Struktur für Portfolio-, Projekt-
oder Fallstudienabgaben im Kurs. Es ist so angelegt, dass es direkt zur
Bewertungslogik in `BEWERTUNGSRUBRIK_PORTFOLIO.md` passt.

Für eine noch schnellere Orientierung vor dem Ausfüllen kann
`KURZGUIDE_PORTFOLIO_STUDIERENDE.md` genutzt werden.

## 1. Titel und Kurzbeschreibung

- Titel des Projekts:
- Name der bearbeitenden Person oder Gruppe:
- Kurze Einordnung in 2 bis 4 Sätzen:

## 2. Fragestellung

Beschreibe knapp und konkret:

- welches Problem bearbeitet wird
- warum das Problem relevant ist
- welches Ziel die Arbeit verfolgt

Leitfrage:

- Was soll am Ende gezeigt, erklärt, vorhergesagt oder verbessert
  werden?

## 3. Datenbasis

Dokumentiere die verwendeten Daten möglichst knapp und nachvollziehbar:

- Datenquelle
- Art der Daten
- Umfang oder Größenordnung
- wichtige Merkmale oder Spalten
- erkennbare Grenzen oder Qualitätsprobleme

Leitfrage:

- Warum sind diese Daten für die Fragestellung geeignet?

## 4. Vorgehen

Beschreibe den Kernworkflow in nachvollziehbaren Schritten:

1. Datenzugriff oder Datenaufbereitung
2. Analyse oder Modellierung
3. Visualisierung, App oder API-Demo
4. Auswertung der Ergebnisse

Wichtig:

- Nicht jede technische Einzelheit auflisten.
- Den logischen Ablauf verständlich machen.

## 5. Technische Umsetzung

Halte fest, was tatsächlich gebaut wurde:

- Notebook, Skript, Streamlit-App, Dashboard oder API
- zentrale Bibliotheken oder Werkzeuge
- wichtigste Datei oder Einstiegspunkt
- kurze Startanleitung

Beispiel:

```bash
pip install -r requirements.txt
streamlit run app.py
```

## 6. Ergebnisse

Dokumentiere die wichtigsten Resultate:

- zentrale Beobachtungen
- Modellgüte, Vergleich oder qualitative Aussage
- ein Plot, Screenshot oder kurzes Tabellenbeispiel
- kurze Interpretation der Ergebnisse

Leitfrage:

- Was ist das wichtigste Ergebnis der Arbeit?

## 7. Grenzen und Reflexion

Diese Sektion ist verpflichtend, auch wenn die Lösung gut funktioniert.

Beschreibe zum Beispiel:

- welche fachlichen oder technischen Grenzen bestehen
- welche Annahmen getroffen wurden
- was bei mehr Zeit verbessert werden könnte
- was an der Lösung robust ist und was nicht

## 8. Eigenleistung

Mache kurz sichtbar, worin die eigene Leistung bestand:

- Auswahl oder Anpassung von Daten
- Modellvergleich oder Variantenentscheidung
- Strukturierung einer App oder eines Dashboards
- eigene Visualisierung, Auswertung oder Dokumentation

Bei Gruppenarbeiten zusätzlich:

- kurze Zuordnung der Teilbeiträge

## 9. Optionale Kürleistungen

Nur angeben, wenn tatsächlich vorhanden:

- Deployment oder öffentliche Bereitstellung
- erweiterte Modellierung
- zusätzliche Tests oder Monitoring-Ideen
- verbesserte Nutzerführung oder Visualisierung
- besondere Erweiterungen über den Pflichtkern hinaus

## 10. Abgabe-Checkliste

Vor der Abgabe prüfen:

- Fragestellung klar formuliert
- Grundlösung lauffähig oder sauber demonstrierbar
- Vorgehen verständlich dokumentiert
- Ergebnisse knapp interpretiert
- Grenzen ehrlich benannt
- Startpunkt der Arbeit klar angegeben

## Kompakte Minimalversion

Wenn die Abgabe bewusst klein gehalten wird, sollten mindestens diese
fünf Bausteine enthalten sein:

1. Problem und Ziel
2. Daten und Vorgehen
3. lauffähiger Kern oder belastbare Demo
4. Ergebnisse
5. Grenzen und Reflexion

## Hinweis für Lehrende

Dieses Template ist bewusst auf Nachvollziehbarkeit statt auf maximale
Länge ausgelegt. Eine kurze, saubere und ehrliche Dokumentation ist für
den Kurs meist wertvoller als eine formal große, aber inhaltlich diffuse
Abgabe.
